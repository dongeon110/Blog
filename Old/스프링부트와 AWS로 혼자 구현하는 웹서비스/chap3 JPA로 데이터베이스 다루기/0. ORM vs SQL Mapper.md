# ORM vs SQL Mapper  
ORM(Object Relational Mapping) vs SQL Mapper  

- - -
## 퍼시스턴스 Persistence  
퍼시스턴스는 영속성이라는 뜻으로 단어처럼 데이터의 지속성을 의미합니다.  
데이터를 생성한 프로그램이 **종료가 되더라도 사라지지 않고** 다시 실행하더라도  
이전에 저장한 데이터를 다시 불러올 수 있습니다.  

## 퍼시스턴스 프레임워크 Persistence Framework
퍼시스턴스 프레임워크를 사용하면 JDBC 프로그래밍의 복잡함이나 번거로움 없이 데이터베이스와 연동되는 시스템 개발을 빠르게 개발 할 수 있습니다.  

퍼시스턴스 프레임워크에는  
SQL 문장으로 직접 DB 데이터를 다루는 **SQL Mapper**와  
자바 객체를 통해 간접적으로 DB 데이터를 다루는 **ORM** 객체 관계 맵퍼가 있습니다.  

SQL Mapper에는 대표적으로 mybatis가 있고,  
객체 관계 맵퍼에는 하이버네이트(Hibernate)가 있습니다.  

# ORM 장단점
## 장점  
1. 특정 DBMS에 종속되지 않는다.  
객체 관계 맵퍼는 프레임워크에서 제공하는 API와 전용 객체 질의어를 사용하여 데이터를 다룹니다.  
프레임워크에서 제공하는 객체 질의어는 SQL보다 단순하여 배우기 쉽고 실행시 DBMS에 맞추어 SQL문을 자동 생성하기 때문에 특정 DBMS에 종속되지 않는 애플리케이션을 만들 수 있습니다.  

2. 생산성이 높다.  
쓰기지연, 변경감지, 지연로딩 등을 제공하며 성능상으로 이점을 얻을 수도 있습니다.  
코드레벨로 관리 되어 사용이 용이하고 생상선이 높습니다.  

3. 객체지향적으로 개발이 가능하다.  
DB는 기본적으로 **데이터를 어떻게 저장할 지**에 초점이 맞춰진 기술입니다.  
하지만, 자바는 **객체지향**으로 추상화, 캡슐화, 정보은닉, 다형성 등 여러 정의들을 다룹니다.  
이렇듯 관계형 데이터베이스와 객체지향 프로그래밍은 서로 패러다임이 불일치합니다.  
객체 관계 맵퍼는 기본적으로 **객체지향적으로 개발**을 하고, 이를 관계형 데이터베이스에 맞게 SQL 대신 생성해서 실행해주기 때문에, SQL에 종속적인 개발을 하지 않아도 됩니다.  

자바 ORM 기술에 대한 API 표준 명세인 JPA(Java Persistence API)도 ORM 하이버네이트(Hibernate)로 구현되었습니다.  

## 단점  
1. 데이터베이스의 정규화가 잘 되어 있어야 한다.  
정규화가 잘 되어 있어야지만 테이블과 객체를 연결하기 쉽고, 객체를 통해 테이블의 데이터를 다루기가 쉽습니다.  
시스템을 개발하는 초기에는 데이터 모델링을 통해 정규과가 잘 되어 있지만, 유지보수 단계에 들어가면 잦은 기능 변경과 추가로 말미암아 초기에 잘 짜여진 데이터베이스의 구조가 흐트러지게 됩니다.  
이렇게 되면 이를 해결하기 위해 각종 편볍을 동원하게 되고, 이 때문에 코드가 복잡해져서 오히려 개발 비용이 증가할 수 있습니다.  

2. 데이터베이스 최적화를 할 수 없다.  
보통 데이터베이스들은 실행 성능을 높이고 데이터 처리를 쉽게 해주는 자신만의 특별한 기능을 가지고 있습니다.  
이런 기능을 활용할 수 있도록 데이터베이스 전용 SQL을 제공합니다.  
하지만 ORM에서는 SQL문을 직접 작성하지 않기 때문에 이런 특장점을 활용하기 어렵습니다.  

# SQL Mapper 장단점  
## 장점  
1. 객체 관계 맵퍼와 달리 SQL문을 감추거나 대체하지 않습니다.  
오히려 개발자가 SQL문을 더 쉽게 작성하고 관리하도록 도와줍니다.  
이를 위해 SQL을 별도의 파일로 분리하였습니다.  
이렇게 되면 SQL의 작성을 데이터베이스 및 업무 전문가에게 맡길 수 있습니다.  

2. 데이터베이스 최적화가 가능합니다.  
ORM 단점 2.번과 같은 이유입니다.  
데이터베이스가 가지고 있는 자신만의 특별한 기능들과 데이터베이스 전용 SQL을 활용할 수 있습니다.  

## 단점  
1. 객체지향적인 개발이 어렵습니다.  
ORM 장점 3.번과 같은 이유입니다.  
객체지향 프로그래밍과 달리, DB는 어떻게 데이터를 저장할 지에 초점이 맞춰진 기술입니다.  
이런 패러다임 불일치 문제로 인해 ORM보다 객체지향적으로 개발을 하기 어렵습니다.  


아직 SI 환경에서는 Spring & MyBatis를 많이 사용하지만,  
자사 서비스를 개발하는 곳에서는 SpringBoot & JPA를 전사 표준으로 사용하고 있습니다.  

그 외 나머지 서비스 기업들 역시 기존 프로젝트 환경을 개편하기는 분들은 대부분 JPA를 선택하고 있습니다.  
그래서 이번 챕터에서는 자바 표준 ORM인 JPA를 사용해보겠습니다.
